# 0002. 에이전트 역할 분담 및 워크플로우

- **상태**: Accepted
- **날짜**: 2026-03-03

## 배경

Claude Code의 서브에이전트를 활용하여 개발 작업을 병렬 처리하되,
각 에이전트가 명확한 책임 범위 안에서 작업해야 혼선이 없다.
초기에는 에이전트가 자기 판단으로 기술 결정을 내리고 칸반 업데이트를 빠뜨리는 문제가 있었다.

## 선택지

1. **단일 에이전트**: 모든 작업을 하나의 에이전트가 순차 처리
2. **역할 기반 분담**: PM → Architecture → BE/FE → QA 파이프라인
3. **자유 분담**: 에이전트가 자율적으로 작업을 가져가는 방식

## 결정

**역할 기반 파이프라인**을 채택한다.

```
PM(기획·칸반 등록) → Architecture(기술 방향·BE/FE 분배) → BE/FE(구현) → QA(검증)
```

### 각 에이전트의 경계

| 에이전트 | 하는 일 | 하지 않는 일 |
|----------|---------|-------------|
| **PM** | 비즈니스 요구사항 정리, PRD 관리, 칸반 등록 | 기술 결정, 파일명·API 경로 언급 |
| **Architecture** | 기술 방향 결정, DTO/Enum 정의, 이슈에 구현 방향 코멘트 | 비즈니스 로직 구현, UI 컴포넌트 작성 |
| **Backend** | NestJS API 구현, 권한 검증, DB 트랜잭션 | FE 코드 수정, Architecture 결정 변경 |
| **Frontend** | Next.js UI 구현, API 연동, 디자인 적용 | BE 코드 수정, Architecture 결정 변경 |
| **QA** | 보안 감사, 테스트 시나리오 작성, 빌드 검증 | 코드 직접 수정 |

### 칸반 업데이트 규칙

- 모든 에이전트는 작업 시작 시 In Progress, 완료 시 Done으로 칸반 상태를 변경한다.
- `--label agent:<역할>` 라벨을 부착하여 누가 작업했는지 추적한다.

## 결과

- PM이 칸반에 등록한 비즈니스 요구사항에 기술 디테일이 섞이지 않음
- Architecture가 중간 다리 역할을 하여 BE/FE가 명확한 방향성을 받고 구현
- QA가 PM의 수용 기준 기반으로 독립 검증 → 기획 의도 누락 방지
